# Урок 15 — Git и работа в команде

## Зачем нужен Git

Git — система контроля версий. Отслеживает все изменения в коде.

**Без Git:**
- Файлы `main_v2_final_FINAL.js` 😱
- Невозможно работать в команде
- Нельзя откатить сломанный код

**С Git:**
- Полная история всех изменений
- Несколько разработчиков работают параллельно
- Легко откатиться к любой предыдущей версии

---

## Основные концепции

### Repository (репозиторий)
Папка проекта под управлением Git. Содержит скрытую папку `.git/` с историей.

```bash
git init           # создать новый репозиторий
git clone URL      # скачать существующий
```

### Commit (коммит)
Снимок состояния кода в определённый момент.

```
История коммитов:
o - "feat: добавить страницу клиентов"   ← последний
|
o - "feat: создать NestJS модуль CRM"
|
o - "feat: начальная структура проекта"   ← первый
```

### Branch (ветка)
Независимая линия разработки. Главная ветка — `main`.

```
main: o---o---o---o
           \
feature:    o---o---o   ← разработка новой функции
```

## Ежедневный рабочий процесс

```bash
# 1. Посмотреть что изменилось
git status

# 2. Добавить файлы для коммита
git add apps/web/app/(dashboard)/clients/page.tsx
git add apps/api/src/modules/clients/

# Или добавить всё изменённое:
git add -A

# 3. Создать коммит
git commit -m "feat(clients): добавить поиск по телефону"

# 4. Отправить на GitHub
git push origin main
```

---

## Ветки — как работать с командой

```bash
# Создать новую ветку для работы над задачей
git checkout -b feature/loyalty-module
# или: git switch -c feature/loyalty-module

# Работаем, коммитим...
git add .
git commit -m "feat(loyalty): добавить начисление баллов"
git commit -m "feat(loyalty): добавить историю транзакций"

# Отправить ветку на GitHub
git push origin feature/loyalty-module

# После проверки (Pull Request) — слить в main
git checkout main
git merge feature/loyalty-module
git push origin main

# Удалить ветку
git branch -d feature/loyalty-module
```

## Соглашение о коммитах

Мы используем **Conventional Commits** — стандарт для сообщений коммитов:

```
тип(область): краткое описание

feat     - новая функциональность
fix      - исправление ошибки
refactor - рефакторинг (не фича, не фикс)
style    - форматирование, без изменения логики
docs     - документация
test     - тесты
chore    - служебные задачи (настройки, deps)
```

Примеры:
```
feat(clients): добавить фильтр по статусу абонемента
fix(rfid): исправить двойной чекин при быстром проходе
feat(crm): реализовать drag-and-drop в Kanban
docs: обновить инструкцию по запуску
refactor(visits): вынести логику чекина в отдельный метод
```

## .gitignore — что НЕ коммитить

Файл `.gitignore` говорит Git игнорировать эти файлы:

```gitignore
# Наши зависимости (тысячи файлов)
node_modules/

# Собранный код
dist/
.next/

# Секреты (НИКОГДА не коммитить!)
.env
.env.local

# Данные SQLite
*.db

# Файлы macOS
.DS_Store
```

**ГЛАВНОЕ ПРАВИЛО:** Никогда не коммитьте файлы `.env` с реальными паролями и ключами!

## Полезные команды

```bash
# Посмотреть историю коммитов
git log --oneline

# Посмотреть изменения
git diff                    # что изменилось (ещё не добавлено)
git diff --staged           # что будет в следующем коммите

# Откатить изменения в файле (осторожно!)
git checkout -- file.txt    # вернуть файл к последнему коммиту

# Последний коммит оказался плохим
git revert HEAD             # создать коммит-отмену (безопасно)

# Посмотреть ветки
git branch -a               # все ветки (локальные + удалённые)

# Обновить с GitHub
git pull origin main

# Сохранить незакоммиченные изменения и переключиться на другую ветку
git stash                   # сохранить
git stash pop               # восстановить
```

## Pull Request (PR) — обзор кода

Pull Request = запрос на слияние ветки в main.

**Процесс:**
1. Разработчик создаёт ветку, работает, пушит на GitHub
2. Открывает Pull Request: "Хочу слить feature/crm-kanban в main"
3. Коллеги проверяют код, оставляют комментарии
4. После одобрения — мержат в main

```bash
# Создать PR через GitHub CLI
gh pr create --title "feat: добавить Kanban в CRM" \
  --body "Реализован drag-and-drop с @dnd-kit. Добавлено перемещение сделок через API."
```
